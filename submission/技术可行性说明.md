# Dendrite · 技术可行性说明
### 参赛赛事：手机上的创意 AI 挑战赛
#### 移动端 AI 创新应用 —— Qwen 系列 + MNN（Arm SME2）+ 端云协同

本说明论证关键能力**均已实现并经真机验证**，并清晰区分「已验证」与「目标/路线图」，不夸大。

---

## 一、系统架构总览

| 层 | 技术选型 | 说明 |
| :--- | :--- | :--- |
| 客户端 | **Flutter**（Android 优先，单代码库） | 移动原生体验；Web 经条件编译复用同一逻辑。 |
| 状态管理 | **flutter_bloc / Cubit** | `ChatCubit` 统一持有数据、持久化与推理流式生命周期。 |
| 本地存储 | **Drift + SQLite** | 知识树以「父指针」存于本地；祖先血脉用递归查询毫秒级重建。 |
| 密钥安全 | **flutter_secure_storage** | 云端 Qwen 的 API Key 存平台安全区，不入明文。 |
| 推理引擎 | **OpenAI 兼容客户端 + 多方言** | 一套 `AgentEngine` 适配 OpenAI / Anthropic / Gemini 方言；本地与云端 Qwen 走同一路径。 |
| 端侧推理 | **MNN 运行时（目标 Arm SME2）** | 本地以 OpenAI 兼容服务承载 Qwen 推理。 |

**数据流：** UI → `ChatCubit` → `AgentEngine.submitQueryStream()`（dart:io 原生 SSE，真正逐 token 流式）→ 流式回填节点 → 落库 Drift → 可序列化为 Markdown。

## 二、Qwen 模型选型（满足「至少一个 Qwen 模型」）

- **端侧（Edge）：Qwen3-1.7B**（MNN 格式，约 1.2 GB 权重）—— 本作落地验证的主力本地模型，质量/速度平衡、多分支分析不空答。
- **轻量端侧：Qwen3-0.6B/0.8B 级** —— 低端机/极速场景。
- **云端协同（Cloud）：Qwen** —— 通过 **DashScope（阿里云百炼）/ 魔搭** 的 **OpenAI 兼容端点**接入（如 `qwen-plus`、`qwen-max`、`qwen3-*`）；在设置里填入 Base URL/Key/模型即可，无需改代码。
- **多模态（路线图）：Qwen3-VL-2B/4B** —— 图文输入，复用同一摄取/分支管线。

> 端、云两侧均为 Qwen，赛题模型要求被双重满足。

## 三、MNN + Arm SME2 加速（赛题推荐技术）

- **为何 MNN：** 轻量、跨端、自带 LLM 运行时（`MNN.llm.create().load()/response()`），并提供面向 **Arm SME2（Scalable Matrix Extension 2）** 的加速后端——SME2 的矩阵乘加单元正是 Transformer 推理的热点，能把端侧吞吐显著拉高，**真正释放设备算力**。
- **接入方式：** 用一个 **OpenAI 兼容的本地服务**（`/v1/chat/completions`，SSE）暴露 MNN 承载的 Qwen 推理；App 只需把「本地后端」Base URL 指向它，**与云端 Qwen 完全相同的客户端代码路径**，零特判。
- **已验证（事实）：**
  - MNN 成功加载并运行 **Qwen3-1.7B**，真实流式产出；
  - 完整链路：Android（WSA）App → 本地 MNN/Qwen 服务 → 流式回包 → 树状渲染；
  - 自动化 `integration_test` 全绿（见第六节数据）。
- **关于 SME2（诚实声明）：** MNN 运行时会做指令集能力探测（启动横幅形如 `i8mm / sve2 / sme2 ...`）。本次验证所用宿主 CPU **不具备 SME2（探测为 0）**，因此 SME2 加速在该环境**未实测启用**；在**具备 SME2 的 Arm 设备**上，MNN 通过后端选择即可启用该加速路径。**架构已为 on-device Arm 推理就绪**（同一 OpenAI 兼容契约），真机 SME2 实测为明确的部署目标（见第七节）。

## 四、端云协同混合架构（已实现）

- **后端模式：** `BackendMode { edge, cloud }`，`ChatState` 同时持有 `localConfig`（Edge/Qwen·MNN）与 `apiConfig`（Cloud/Qwen）。
- **默认后端可切换：** 顶栏芯片实时切换 Edge↔Cloud；设置中心可配默认后端、本地服务地址与模型；配置持久化（`SettingsRepository.loadEdge/saveEdge`）。
- **按分支路由（per-branch router）：** 每条分支可单独选「跟随默认 / 本地 / 云端」——`createBranchQuestion(..., backend:)` → `_triggerAICompletion` 按 `state.configFor(mode)` 解析引擎。**主干本地展开、关键分支上云**成为现实。
- **本地可执行保证：** 提问、分支、内容摄取、导出等**核心交互逻辑全部可在端侧 Qwen 上跑通**，云端为按需增强。

## 五、内容摄取层（arXiv / GitHub，已实现）

- `UrlDetector` 识别 arXiv（`/abs`、`/pdf`、旧式 id）与 GitHub（repo / `tree` / `blob` / raw）。
- `ArxivFetcher`：arXiv Atom API → 标题/摘要/作者/分类 → 概述主干 + 5 条分析分支种子。
- `GithubFetcher`：REST 取仓库描述 + README + 文件树；或 raw CDN 取单文件 → 概述 + 分支种子；支持 Bearer Token 提限额（raw CDN 无限额）。
- `ChatCubit.ingestUrl()`：每次分析**新开一棵树**（概述为真根），逐条 `createBranchQuestion` 散开，全部由当前后端（默认端侧 Qwen）作答。

## 七、性能、资源、部署与隐私

- **模型体量：** Qwen3-1.7B-MNN ≈ 1.2 GB 权重；0.6B/0.8B 级更小，适配低端机。
- **延迟：** 当前在**宿主 CPU（无 SME2）**上单次分支作答约 14–58 s；在 **SME2/NPU 加速的 Arm 真机**上预期显著下降——这正是采用 MNN(SME2) 的核心收益。
- **线程安全：** 多分支串行 + 服务端加锁，保证 MNN 单实例安全。
- **部署：** Android Release 可构建；自适应图标、网络安全配置（本地 HTTP 仅按内网网关放行，生产强制 HTTPS）就绪。iOS 代码兼容，待 Mac/证书补齐。
- **隐私与成本：** 端侧 Qwen → 数据不出端、离线可用、边际推理成本趋零。

## 八、已知限制与路线图

- **真机 Arm SME2 实测：** 将 MNN/Qwen 以**嵌入式运行时（AAR/JNI）或设备内服务**部署到具备 SME2 的 Arm 手机，实测吞吐与功耗（首要目标）。
- MNN 当前 build 不响应 `max_new_tokens`（仅按 EOS 停止）；Qwen3 思考模式较冗长，UI 已将 `<think>` 与正文分离展示。
- 路线图：端侧 NPU 协同、Qwen3-VL 图文摄取、PDF/网页摄取、按分支按复杂度/成本/隐私的自动 local/cloud 路由。

> 结论：基于 Qwen 的端云协同、MNN 端侧推理、内容摄取与闭环导出均已落地并经真机验证；以 MNN（Arm SME2）释放设备算力的技术路径清晰、可行、可持续演进。
